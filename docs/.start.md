# Start Here: Modern Unix, Nushell, and Nix

This guide uses this repository itself as the practice target for:

- modern Unix commands
- Nushell
- Nix, `nix-darwin`, and Home Manager

Unless noted otherwise, run everything from the repository root.

## Big Picture

The main files in this repository are:

- entry point: [flake.nix](../flake.nix)
- macOS system layer: [modules/darwin/core.nix](../modules/darwin/core.nix)
- GUI app exposure: [modules/darwin/desktop-apps.nix](../modules/darwin/desktop-apps.nix)
- Home Manager user layer: [home/default.nix](../home/default.nix)
- shell setup: [home/shell.nix](../home/shell.nix)
- Git and `delta`: [home/git.nix](../home/git.nix)
- initial bootstrap: [scripts/bootstrap-macos.sh](../scripts/bootstrap-macos.sh)
- apply changes: [scripts/apply.sh](../scripts/apply.sh)
- verify health: [scripts/doctor.sh](../scripts/doctor.sh)

Script policy in this repository:

- write repo shell scripts in Nushell
- keep `scripts/bootstrap-macos.sh` in POSIX `sh` because it must run before Nushell is available

The normal workflow in this setup is:

1. Explore with `eza`, `fd`, `rg`, and `bat`
2. Process structured data with Nushell
3. Persist changes in Nix config
4. Apply with `./scripts/apply.sh`
5. Verify with `./scripts/doctor.sh`

## 0. Setup

Move into the repository:

cd /path/to/origin

## 1. Modern Unix Hands-on

### 1-1. List files and understand structure

```bash
eza -lah --git
eza --tree --level=2 home
```

What to notice:

- `eza` is a nicer `ls`
- `--git` shows Git status inline
- `--tree` is great for quick structure discovery

This setup also defines these aliases:

- `l` -> `eza -lah --git`
- `lg` -> `eza -lah --git`
- `lt` -> `eza --tree --level=2`

Those aliases live in [home/default.nix](../home/default.nix).

### 1-2. Find files

```bash
fd -e nix . home
fd shell home
```

What to notice:

- `fd` is usually much simpler than `find`
- extension and name searches read naturally

### 1-3. Search text

```bash
rg 'programs\.' home
rg 'delta|git' home/git.nix
```

What to notice:

- `rg` is the default codebase search tool here
- it is a much better starting point than recursive `grep`

### 1-4. Read files

```bash
bat home/shell.nix --paging=never
bat home/default.nix --paging=never
```

What to notice:

- `bat` is a better `cat` for source files
- syntax highlighting makes config review much easier

In this setup, `cat` is aliased to `bat`.

### 1-5. Disk, processes, and HTTP

```bash
dust -d 1
duf
procs
btm
xh get https://httpbin.org/json
```

What to notice:

- `dust` is a friendlier `du`
- `duf` is a friendlier `df`
- `procs` is easier to read than `ps`
- `btm` gives you a useful terminal dashboard
- `xh` is a cleaner alternative to `curl`

Notes:

- quit `btm` with `q`
- this setup aliases `du -> dust` and `df -> duf`

### 1-6. Replace text and inspect diffs

```bash
printf 'home-manager\nhome shell\n' | sd 'home' 'HOME'
git diff -- home/shell.nix
```

What to notice:

- `sd` is excellent for simple replacements
- `git diff` is rendered through `delta` in this setup

The `delta` integration is configured in [home/git.nix](../home/git.nix).

### 1-7. Use the Git shortcuts in this setup

This repository has two layers of Git shortcuts:

- shell aliases in `home/default.nix`, such as `gs` and `gd`
- Git subcommand aliases in [home/git.nix](../home/git.nix), such as `git ap` and `git last`

For day-to-day work, prefer the shell aliases where they exist.

Try these first:

```bash
gs
gd
ga home/shell.nix
gm "test commit message"
git ds
git last
```

Common shell shortcuts from this setup:

- `g` -> `git`
- `ga` -> `git add`
- `gaa` -> `git add --all`
- `gam` -> `git commit --amend`
- `gb` -> `git branch`
- `gbda` -> `git gbda`
- `gco` -> `git checkout`
- `gd` -> `git diff`
- `gf` -> `git fetch`
- `gl` -> `git pull`
- `gm` -> `git commit -m`
- `gp` -> `git push`
- `gs` -> `git status -sb`
- `gsw` -> `git switch`

Useful Git-only aliases from this setup:

- `git ap` -> `git add -p`
- `git ds` -> `git diff --staged`
- `git grm` -> `git rm -rf --cached`
- `git last` -> `git log -1 HEAD --stat`
- `git lg` -> `git log --graph --decorate --oneline --all`
- `git rb` -> `git rebase`
- `git ri` -> `git rebase -i`
- `git unstage` -> `git restore --staged`

One detail to remember:

- `lg` is already a shell alias for `eza -lah --git`, so use `git lg` for the Git log alias
- `gbda` deletes all local branches except the current one, so treat it as a cleanup command

### 1-8. Clone into the workspace layout

This setup includes a `clone` helper that keeps GitHub and GitLab repositories under the
expected `~/Source/<host>/<owner-or-group>/...` layout.

Try the built-in help first:

```bash
clone --help
```

Then try a real clone:

```bash
clone github your-user/origin tutorial
pwd
eza -lah ~/Source/github.com/your-user
```

What to notice:

- `clone github your-user/origin tutorial` clones over SSH
- the local directory becomes `~/Source/github.com/your-user/origin--tutorial`
- the extra `tutorial` argument is an alias, so you can keep multiple local clones of the same repo
- if you omit the alias, the directory name stays as the repository name

You can use the same flow with GitLab:

```bash
clone gitlab my-group/platform/api qa
clone git@gitlab.com:my-group/platform/api.git sandbox
```

Important behavior:

- only `github.com` and `gitlab.com` are supported
- only SSH inputs are accepted, so HTTPS remotes are rejected
- the command stops if the destination path already exists

The implementation lives in [scripts/clone.sh](../scripts/clone.sh), and the workspace
layout is documented in [workspace.md](./workspace.md).

## 2. Nushell Hands-on

This machine uses Nu as the login shell. The relevant setup is in
[modules/darwin/core.nix](../modules/darwin/core.nix)
and
[home/shell.nix](../home/shell.nix).

### 2-1. Enter Nu

```bash
nu
```

Run the rest of this section inside Nushell.

### 2-2. Nu pipelines work on tables

```nu
ls home | select name type size
ls home | where type == file | sort-by size | reverse
```

What to notice:

- `ls` returns structured data, not plain text
- `where`, `select`, and `sort-by` operate directly on that data

### 2-3. Parse and query JSON directly

```nu
open flake.lock | from json | get nodes | columns
open flake.lock | from json | get nodes.nixpkgs.locked.rev
```

What to notice:

- `open` gives you file contents
- `from json` turns JSON text into structured Nu values
- `get` lets you pull nested fields directly

### 2-4. Turn text into a list

```nu
open README.md | lines | where $it =~ "modern Unix"
```

What to notice:

- text data usually needs `lines` before list-style processing

### 2-5. Connect external commands to Nu

```nu
^fd -e nix . home | lines | path basename
^rg -n 'programs\.' home | lines | first 5
```

What to notice:

- `^` explicitly calls an external command
- external command output is still plain text until you convert it

### 2-6. Build small data transformations

```nu
let tools = [bat eza fd rg]
$tools | each {|tool| {tool: $tool, path: (which $tool | get path.0)} }
```

What to notice:

- `let` binds values
- `each` maps over a list
- records become readable tables automatically

### 2-7. Exit Nu

```nu
exit
```

### 2-8. The main Nu mindset

- know whether your pipeline is carrying text, a list, a record, or a table
- `ls` already gives you structured values
- `open ... | from json` is the standard JSON pattern
- external commands usually need a conversion step like `lines`

## 3. Nix Hands-on

This repository is flake-based. Start by seeing what the flake exports.

```bash
nix flake show --no-write-lock-file .
```

Expected takeaway:

- the main output here is `darwinConfigurations.workstation`
- machine-specific values come from `machine/local.env` when you use the repo scripts

### 3-1. Read the flake

```bash
bat flake.nix --paging=never
```

What to notice:

- `inputs` contains `nixpkgs`, `nix-darwin`, and `home-manager`
- `outputs` builds `darwinConfigurations.workstation`
- system modules and user modules are composed together

### 3-2. Read the system layer

```bash
bat modules/darwin/core.nix --paging=never
bat modules/darwin/desktop-apps.nix --paging=never
```

What to notice:

- the login shell is `nushell`
- `environment.shells` includes `bash`, `nu`, and `zsh`
- macOS defaults are managed declaratively
- GUI apps are exposed into `/Applications`

### 3-3. Read the user layer

```bash
bat home/default.nix --paging=never
bat home/shell.nix --paging=never
bat home/git.nix --paging=never
```

What to notice:

- `home.packages` defines user-facing CLI tools
- `home.shellAliases` defines the shell aliases
- `programs.git.settings.alias` defines the Git subcommand aliases
- `programs.nushell`, `programs.fzf`, `programs.zoxide`, and `programs.tmux` live here
- `programs.delta.enableGitIntegration = true` wires `delta` into Git

### 3-4. Evaluate concrete values

```bash
nix eval --raw .#darwinConfigurations.workstation.config.networking.hostName
nix eval --raw .#darwinConfigurations.workstation.config.system.primaryUser
nix eval --raw .#darwinConfigurations.workstation.config.users.users.$(nix eval --raw .#darwinConfigurations.workstation.config.system.primaryUser).shell.name
nix eval --json .#darwinConfigurations.workstation.config.home-manager.users.$(nix eval --raw .#darwinConfigurations.workstation.config.system.primaryUser).home.sessionVariables | jq
```

What to notice:

- `nix eval` lets you inspect final values directly
- you can confirm your understanding of the module graph by evaluating it

### 3-5. Day-to-day operational commands

Initial bootstrap:

```bash
./scripts/bootstrap-macos.sh
```

Apply config changes:

```bash
./scripts/apply.sh
```

Run a health check:

```bash
./scripts/doctor.sh
```

What each script does:

- `bootstrap-macos.sh`: initial machine setup when Nix is not ready yet
- `apply.sh`: runs `darwin-rebuild switch --flake ".#workstation" --impure`
- `doctor.sh`: checks whether the key tools are available

4. explain the role of each major Nix file in this repository
5. make a config change, apply it, and verify it
